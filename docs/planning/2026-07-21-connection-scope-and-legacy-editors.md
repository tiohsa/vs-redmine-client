# 接続スコープ固定と旧形式編集ファイル対応

日付: 2026-07-21
ステータス: 完了
プロジェクトルート: `/home/glorydays/projects/src/ts/pr`

## 依頼内容

新規チケット・新規コメント作成の開始時接続スコープを非同期処理全体で固定し、接続切替時の誤送信を防ぐ。旧形式編集ファイルは安全な明示割当で同期可能にし、回帰テスト、CI、バージョン、PR状態を整える。

## 背景

`syncToRedmine.ts` は開始時の `operationScope` を取得しているが、通常の新規下書き生成経路は登録時の現在スコープを使っていた。`ticketEditorRegistry.ts` は接続ハッシュを持たない編集ファイルを `unresolved:unknown` として登録し、明示割当後に同期可能とする。

## 調査結果

- 新規チケットは `syncNewTicketDraft()` から `rewriteNewTicketEditorToTicketMode()` を呼び、登録時のスコープ伝播を確認する必要がある。
- 新規コメントは `syncNewCommentDraft()` の `onCreated` から `finalizeNewCommentDraftDocument()` を呼ぶ。
- `resolveEditorConnectionScope()` は旧形式ファイルに `unresolved:unknown` を返す。
- CodeGraph は未初期化だったため、既存ソースとテストを根拠に実装する。
- 初回は依存関係未導入で型解決に失敗したが、依存関係導入後に再検証した。

## 判断

- 旧形式ファイルは接続先不明として同期を止め、現在接続先への明示割当コマンドを提供する。
- 接続切替後も開始時スコープで状態を更新し、現在接続との不一致は既存の同期ガードに委ねる。

## 実装計画

1. 旧形式スコープ解決を未解決状態にし、明示割当コマンドを追加する。
2. 新規作成処理のスコープ伝播を確認・修正し、接続切替回帰テストを追加する。
3. バージョン更新とCI検証を行い、計画結果を追記する。

## リスクと対策

- リスク: 接続切替後に既存接続のエディターが現在接続へ誤登録される。対策: 開始スコープを明示引数で伝播し、テストで登録値を検証する。
- リスク: 依存関係未導入でCI検証できない。対策: 依存関係を導入してから全検証を再実行する。

## 検証方法

- `pnpm test`
- `pnpm run lint`
- `pnpm run compile`
- 旧形式および異なる接続ハッシュの登録テスト

## 結果

新規チケット作成（Markdownヘッダー・通常の下書き経路を含む）と新規コメント作成の開始スコープ固定、APIキーの処理開始時スナップショット、旧形式ファイルの明示割当、回帰テスト、`0.5.4` 更新を実装した。`pnpm test` は759件すべて成功し、lint・compileも成功した。Release workflowにもテストを追加した。
